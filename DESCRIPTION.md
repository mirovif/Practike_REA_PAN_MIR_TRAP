# Аналитический веб-сервис на базе датасета Instacart

## 1. Общее описание проекта

Проект — это полноценный аналитический веб-сервис, построенный на реальном датасете Instacart (история онлайн-заказов продуктов питания). Пользователь вводит свой ID покупателя и получает персонализированный анализ: что чаще всего заказывал, как часто делал покупки, какие товары забыл, когда обычно заказывает. Параллельно работает административный дашборд с агрегированной статистикой по всем покупателям.

Стек технологий: **Python + Flask** (веб-сервер), **SQLite** (база данных Instacart), **Mistral AI** (языковая модель для генерации рекомендаций), **Loginom** (аналитические REST-сервисы партнёра), **Chart.js** (интерактивные диаграммы), **GitHub** (система контроля версий).

## 2. Что требовалось по методологии

### 2.1 Подключение к веб-сервисам Loginom

По заданию необходимо интегрироваться с REST-сервисами, которые обрабатывают данные Instacart на стороне Loginom.

Реализовано 4 сервиса:

| Сервис | URL | Что возвращает |
|---|---|---|
| История покупок | `GetUserHistory` | Топ-10 товаров пользователя с кол-вом заказов |
| Ритм покупок | `dop_model_3` | Среднее / мин / макс дней между заказами |
| Забытые товары | `dop_model_4` | Товары, которые не заказывались последние 3 раза |
| Лучшее время | `dop_model` | Тепловая карта заказов по дням и часам |

Все четыре сервиса вызываются через единую функцию `_loginom_get()` в `business_rules.py`. Функция умеет разбирать разные форматы ответа: и чистый список `[...]`, и обёртку `{"DataSet": {"Rows": [...]}}`, которую использует Loginom.

```python
# business_rules.py, строка 113
def _loginom_get(url: str, user_id: int):
    resp = requests.get(url, params={"user_id": user_id}, timeout=30)
    data = resp.json()
    # Разбираем DataSet.Rows или чистый список
    if isinstance(data, dict) and "DataSet" in data:
        return data["DataSet"].get("Rows", [])
    ...
```

### 2.2 Использование языковой модели (Mistral AI вместо GigaChat)

По заданию — интеграция с языковой моделью. Использован **Mistral AI** (`mistral-small-latest`), так как GigaChat недоступен вне России без VPN. Все ключи хранятся только в `.env` файле, который не коммитится в git.

Mistral вызывается в 5 местах:

- Комментарий к ритму покупок (`ask_mistral_rhythm`)
- Напоминание о забытых товарах (`ask_mistral_forgotten`)
- Совет по лучшему времени заказа (`ask_mistral_best_time`)
- Бизнес-аналитика дашборда (`ask_mistral_dashboard`)

## 3. Что придумали сами

### 3.1 Постоянный SQLite-кеш с TTL

**Проблема:** каждый запрос к Loginom и Mistral — это сетевой вызов на 2–10 секунд. Если пользователь обновляет страницу, он ждёт заново.

**Решение:** отдельный файл `cache.db` (SQLite), где хранятся результаты всех запросов с временем истечения.

```
Таблица cache:
  key        TEXT  — идентификатор (например, "history_42")
  value      TEXT  — данные в формате JSON
  expires_at REAL  — unix-timestamp, когда кеш устаревает
```

При каждом запросе сначала проверяется `expires_at > now()`. Если кеш свежий — данные берутся из него (мгновенно), к Loginom и Mistral обращения нет.

TTL разные для разных данных:
- История покупок (Loginom): **5 минут**
- Рекомендации Mistral: **10 минут**
- Дашборд (тяжёлые SQL): **30 минут**

Кеш переживает перезапуски Flask, так как это файл на диске, а не переменная в памяти.

```python
# business_rules.py, строка 64
def _cache_get(key: str):
    row = conn.execute("SELECT value, expires_at FROM cache WHERE key = ?", (key,)).fetchone()
    if row and time.time() < row[1]:   # expires_at не истёк
        return json.loads(row[0]), True  # (данные, hit=True)
    return None, False
```

### 3.2 Система уровней клиента

**Идея:** показать покупателю его «статус» в магазине по аналогии с программами лояльности.

Уровень считается по суммарному количеству заказов по всем товарам в топ-10:

| Уровень | Условие | Значок |
|---|---|---|
| Новичок | до 20 заказов | 🌱 |
| Постоянный | 20–59 заказов | ⭐ |
| Лояльный | 60–119 заказов | 🔥 |
| VIP | 120+ заказов | 👑 |

Уровень рассчитывается в `get_client_level()` на основе данных с Loginom, сохраняется в Flask-сессию и отображается и на `/history`, и на главной странице `/`.

```python
# business_rules.py, строка 209
def get_client_level(rows):
    total = sum(r.get("order_count", 0) for r in rows)
    if total < 20:   return {"level": "Новичок",    "emoji": "🌱", ...}
    if total < 60:   return {"level": "Постоянный", "emoji": "⭐", ...}
    if total < 120:  return {"level": "Лояльный",   "emoji": "🔥", ...}
    return           {"level": "VIP",               "emoji": "👑", ...}
```

### 3.3 Блок «Вас может заинтересовать» с двуязычным промптом

**Идея:** персональные рекомендации конкретных товаров, а не просто слова.

**Техническая сложность:** Mistral хорошо знает названия товаров по-русски, но база данных Instacart — на английском. Нужен «мост».

**Решение — двуязычный промпт:**

1. Mistral получает задачу ответить строго в формате `Русское название / English keyword`
2. Русское название показывается пользователю как «ключевое слово»
3. Английское слово используется для поиска реального товара в SQLite

```python
# business_rules.py, строка 253
user_message = (
    "Порекомендуй ровно 3 товара в формате нумерованного списка:\n"
    "1. Русское название / English keyword\n"
    "2. Русское название / English keyword\n"
    ...
)
# После ответа парсим "/" и ищем в БД
cur = conn.execute("SELECT product_name, aisle FROM products WHERE product_name LIKE ?",
                   (f"%{en_name}%",))
```

Итог: пользователь видит «Яблоки» → «Organic Fuji Apple» (реальный товар из базы) → отдел → кнопка «Добавить в корзину».

Над карточками отображаются чипы с русскими ключевыми словами, по которым шёл поиск.

### 3.4 Административный дашборд с оптимизацией SQL

**Идея:** аналитика по всем покупателям сразу — для «менеджера магазина».

4 блока статистики:
- Топ-10 самых покупаемых товаров (таблица + горизонтальный бар Chart.js)
- Распределение клиентов по уровням (таблица + пончиковая диаграмма)
- Средний интервал между заказами (блоки-метрики)
- Категории с высокой долей повторных покупок

**Проблема производительности:** таблица `orders_prior` содержит 32 миллиона строк. Первая версия запроса по категориям выполнялась **107 секунд** — страница не загружалась вовсе.

**Решение — двухшаговый запрос + покрывающий индекс:**

```sql
-- Медленно (107 сек): JOIN по 32М строкам для получения aisle
SELECT p.aisle, COUNT(*), AVG(op.reordered)
FROM orders_prior op JOIN products p ON op.product_id = p.product_id
GROUP BY p.aisle;

-- Быстро (2.4 сек): сначала GROUP BY product_id (индекс), потом JOIN к 49К строкам
SELECT p.aisle, SUM(sub.total), SUM(sub.reordered_sum)/SUM(sub.total)
FROM (
    SELECT product_id, COUNT(*) AS total, SUM(reordered) AS reordered_sum
    FROM orders_prior GROUP BY product_id   -- 49 688 уникальных product_id
) sub JOIN products p ON sub.product_id = p.product_id
GROUP BY p.aisle;
```

Дополнительно создан покрывающий индекс `(product_id, reordered)` — агрегация идёт только по индексу, без обращения к основным данным таблицы. Ускорение: **в 44 раза**.

### 3.5 Анимированный оверлей загрузки с циклическими текстами

**Идея:** вместо белого экрана при долгом запросе — красивый прелоадер с пояснением, что происходит.

При отправке формы поверх страницы показывается тёмный overlay с вращающимся спиннером и текстом, который циклически меняется каждые 1.8 секунды:

```javascript
// base.html
function showOverlay(steps) {

    _stepTimer = setInterval(() => {

```

На каждой странице — свои тексты:

- `/insights`: «Анализируем ритм покупок...» → «Ищем забытые товары...» → «Определяем лучшее время...»

### 3.6 Удаление приветствий из ответов Mistral

**Проблема:** Mistral начинает ответы с «Здравствуйте!» или «Добрый день!», что выглядит неестественно внутри карточки страницы.

**Решение:** функция `_strip_greeting()` проверяет первое предложение на наличие слов-приветствий и отрезает его, если находит.

```python
# business_rules.py, строка 137
_GREETINGS = ["здравствуйте", "добрый день", "привет", "уважаемый", ...]

def _strip_greeting(text):
    for sep in (". ", "! "):
        idx = text.find(sep)
        if idx != -1 and any(g in text[:idx].lower() for g in _GREETINGS):
            return text[idx + len(sep):].lstrip()
    return text
```

### 3.7 Кеш-бейдж и CTA-кнопки

На страницах `/history` и `/dashboard` рядом с заголовком отображается бейдж **«из кэша»** (синяя таблетка), когда данные взяты из кеша, а не запрошены заново.

Под блоком с рекомендацией Mistral — три действия:
- **📋 Копировать** — копирует текст рекомендации в буфер обмена через `navigator.clipboard`
- **🔄 Обновить анализ** — сбрасывает кеш (`/cache/clear`) и перезапускает анализ
- **⬇️ Сохранить PDF** — вызывает `window.print()` (страница адаптирована под печать: скрываются навигация, формы, кнопки)

### 3.8 Тепловая карта активности (Сервис 3)

Данные от Loginom возвращают пары `(day_of_week, hour, count)`. На их основе JavaScript строит HTML-таблицу 7×N (дни × часы), где интенсивность цвета ячейки пропорциональна количеству заказов:

```javascript
const alpha = maxVal ? (v / maxVal * 0.8 + 0.05) : 0.05;
html += `<td style="background:rgba(49,130,206,${alpha})">${v}</td>`;
```

Самая активная ячейка — непрозрачно-синяя, нулевые — почти прозрачные.

**Поток данных для `/history`:**
```
Браузер → POST /history
  → get_user_history(user_id)
      → cache_get("history_42") → HIT → вернуть из cache.db
      → MISS → GET loginom/.../GetUserHistory?user_id=42
             → cache_set("history_42", данные, TTL=300)
  → ask_mistral(user_id, products_text)
      → cache_get("mistral_42") → HIT → вернуть из cache.db
      → MISS → Mistral API → _strip_greeting() → cache_set(...)
  → get_client_level(rows) → уровень клиента
  → render_template("history.html", ...)
```
